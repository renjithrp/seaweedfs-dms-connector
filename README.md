# dms-seaweedfs-connector

Go implementation of the DMS binary-content connector. It preserves the existing DMS REST contract while storing content in SeaweedFS through filer HTTP endpoints.

## Endpoints

- `GET|POST|PUT /dss/api/put/{tenant_id}`
- `GET /dss/api/ping`
- `GET /dss/api/stats/{file_id}`
- `GET /dss/api/get/{file_id}`
- `GET /dss/api/getimage/{file_id}/{page_number}?scale=1.0`
- `GET /metrics`
- `GET /healthz`
- `GET /readyz`

## Storage Layout

Objects are stored under `SEAWEEDFS_INTERNAL_ROOT`:

- binary content: `/<SEAWEEDFS_INTERNAL_ROOT>/data/{file_id}`
- metadata JSON: `/<SEAWEEDFS_INTERNAL_ROOT>/meta/{file_id}.json`
- rendered PDF previews: `/<SEAWEEDFS_INTERNAL_ROOT>/images/{file_id}_page_{page}.png`
- quarantined binaries when metadata write fails: `/<SEAWEEDFS_INTERNAL_ROOT>/failed/{file_id}__{original_filename}`

This keeps the northbound API stable without requiring a separate metadata database.

### Directory Structure

For the default root `SEAWEEDFS_INTERNAL_ROOT=/connector/dms`, the effective layout looks like this:

```text
/connector/dms/
├── data/
│   └── {file_id}
├── meta/
│   └── {file_id}.json
├── images/
│   └── {file_id}_page_{page}.png
└── failed/
    └── {file_id}__{original_filename}
```

- `data/` stores the original uploaded binary.
- `meta/` stores the metadata record used by `stats`, `get`, and `getimage`.
- `images/` stores cached PNG previews for PDF pages.
- `failed/` stores binaries that were uploaded successfully but could not be paired with metadata.

### Metadata Schema

Each metadata file under `meta/{file_id}.json` is stored as JSON with this structure:

```json
{
  "file_id": "2516583547427874372",
  "tenant_id": "test-tenant",
  "storage_path": "/connector/dms/data/2516583547427874372",
  "size": 49672,
  "content_type": "application/pdf",
  "uploaded_at": "2026-04-17T03:00:00Z",
  "tags": {
    "tenant_id": "test-tenant",
    "original_filename": "sample.pdf"
  }
}
```

Field details:

- `file_id`: generated numeric object ID returned by the upload API
- `tenant_id`: tenant from `/dss/api/put/{tenant_id}`
- `storage_path`: SeaweedFS filer path for the stored binary
- `size`: uploaded object size in bytes
- `content_type`: content type from the incoming request, or detected from file bytes when missing
- `uploaded_at`: UTC timestamp written by the connector
- `tags.tenant_id`: repeated tenant marker for convenience
- `tags.original_filename`: original uploaded file name

## Runtime Behavior

- Uploads stage request content to a local temp file, then stream that file to SeaweedFS.
- IDs are generated by a Snowflake-style numeric generator and require a unique `WORKER_ID` per running instance.
- `/dss/api/ping` returns success only when SeaweedFS is reachable and the ID generator has not detected a backward clock jump.
- `/readyz` checks SeaweedFS reachability.
- `getimage` returns original bytes for image content, and PDF previews for PDF content.
- PDF previews are cached back into SeaweedFS under `images/`.
- If binary upload succeeds but metadata upload fails, the binary is moved into `failed/` as a best-effort quarantine path.

## Metrics

The connector exposes Prometheus-style metrics at `GET /metrics`.

### Generic Request Metrics

- `dms_connector_requests_total{endpoint,method,status}`
  Total request count by logical endpoint, HTTP method, and response status.
- `dms_connector_request_duration_seconds_sum{endpoint,method}`
  Sum of observed request durations by endpoint and method.

### Health Metrics

- `ping_success_count`
  Number of successful `/dss/api/ping` checks.
- `ping_failure_count`
  Number of failed `/dss/api/ping` checks.
- `seaweedfs_connection_fail_count`
  Number of SeaweedFS connectivity-related failures observed by the connector.

### Upload Metrics

- `seaweedfs_upload_success_count{bucket_name}`
  Successful SeaweedFS writes performed by the connector.
- `seaweedfs_upload_fail_count{bucket_name}`
  Failed SeaweedFS writes performed by the connector.
- `file_upload_success_count{bucket_name}`
  Successful DMS upload requests that completed file and metadata creation.
- `file_upload_fail_count{bucket_name,object_name}`
  Failed DMS upload requests keyed by object ID.

### Download Metrics

- `seaweedfs_connection_get_stream_success_count{bucket_name}`
  Successful SeaweedFS download stream opens.
- `seaweedfs_connection_get_stream_count{bucket_name,object_name}`
  Failed SeaweedFS download stream opens.
- `file_download_success_count{bucket_name}`
  Successful DMS file download responses.
- `file_download_fail_count{bucket_name,object_name}`
  Failed DMS file download responses.

### Metadata and Preview Metrics

- `get_stats_success_count{bucket_name}`
  Successful `stats` lookups.
- `get_stats_fail_count{bucket_name,object_name}`
  Failed `stats` lookups.
- `get_pdf_image_success_count{bucket_name}`
  Successful PDF-to-image responses.
- `get_pdf_image_fail_count{bucket_name,object_name}`
  Failed PDF-to-image responses.

## Requirements

- SeaweedFS filer endpoint(s)
- Go toolchain for local builds
- `pdftoppm` for PDF preview generation

On macOS, install PDF tooling with:

```bash
brew install poppler
```

Linux (Debian/Ubuntu)

```bash
sudo apt update
sudo apt install -y poppler-utils
```

## Configuration

Required:

- `SEAWEEDFS_FILERS=http://10.5.93.213:8888,http://10.5.93.214:8888`

Optional:

- `LISTEN_ADDR=:9082`
- `READ_TIMEOUT=30s`
- `WRITE_TIMEOUT=5m`
- `IDLE_TIMEOUT=60s`
- `SEAWEEDFS_INTERNAL_ROOT=/connector/dms`
- `SEAWEEDFS_HTTP_TIMEOUT=60s`
- `TEMP_DIR=/tmp`
- `DOWNLOAD_CACHE_DIR=/tmp/dms-download-cache`
- `DOWNLOAD_CACHE_MAX_FILES=128`
- `MAX_UPLOAD_BYTES=67108864`
- `PDF_RENDER_DPI=144`
- `WORKER_ID=0`

`WORKER_ID` must be unique per connector instance and must be in the range `0..1023`.

## Build

```bash
go build -o bin/dms-seaweedfs-connector ./cmd/server
```

## Test

There are no Go unit tests in the repo right now, so the main verification flow is:

1. Build the binary
2. Start SeaweedFS locally
3. Run the connector
4. Run the smoke test

Build:

```bash
go build -o bin/dms-seaweedfs-connector ./cmd/server
```

Start the local SeaweedFS stack:

```bash
docker compose -f seaweedFS/docker-compose.yaml up -d
```

Run the connector:

```bash
export SEAWEEDFS_FILERS=http://127.0.0.1:8888
export WORKER_ID=0
./bin/dms-seaweedfs-connector
```

Run the smoke test in another terminal:

```bash
BASE_URL=http://127.0.0.1:9082 ./examples/curl-smoke-test.sh
```

Optional health checks:

```bash
curl http://127.0.0.1:9082/dss/api/ping
curl http://127.0.0.1:9082/readyz
curl http://127.0.0.1:9082/metrics
```

## Local SeaweedFS Cluster

A local multi-master SeaweedFS compose stack is available at [seaweedFS/docker-compose.yaml](seaweedFS/docker-compose.yaml).

Start it with:

```bash
docker compose -f seaweedFS/docker-compose.yaml up -d
```

View container status:

```bash
docker compose -f seaweedFS/docker-compose.yaml ps
```

View logs:

```bash
docker compose -f seaweedFS/docker-compose.yaml logs -f
```

Stop it with:

```bash
docker compose -f seaweedFS/docker-compose.yaml down
```

Reset local SeaweedFS data volumes:

```bash
docker compose -f seaweedFS/docker-compose.yaml down
rm -rf seaweedFS/data/*
```

## Performance Test

A `k6` upload test is available at [examples/k6-upload-perf.js](examples/k6-upload-perf.js).

Create a sample file first:

```bash
mkdir -p /tmp/dms-perf
mkfile 25m /tmp/dms-perf/25mb.bin
```

Then run:

```bash
BASE_URL=http://127.0.0.1:9082 \
TENANT_ID=perf-tenant \
FILE_PATH=/tmp/dms-perf/25mb.bin \
FILE_NAME=25mb.bin \
VUS=20 \
DURATION=1m \
k6 run examples/k6-upload-perf.js
```

If `k6` is not available, parallel `curl -F "bin=@..."` requests also work for basic upload benchmarking.
